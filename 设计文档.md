# AI需求分析与测试用例生成平台设计文档（优化版）

## 1. 背景与目标

计划建设一个网页端后台系统，用于辅助产品、测试、研发团队处理需求文档。系统前期可以先通过 skills 或脚本方式跑通核心链路，调用 AI 接口完成自动分析与生成，后续再逐步产品化。

系统主要解决两个问题：

1. 导入产品需求文档后，自动识别需求中不清晰、遗漏、矛盾、边界条件缺失、验收标准不足等问题。
2. 基于产品需求自动生成测试文档，支持功能测试用例、UAT 验收用例，并尽量支持 XMind 思维导图格式导出。

核心设计原则：

1. 文档解析是基础能力，必须先将 Word、PDF、图片、HTML 等多格式输入转成稳定、可追溯的结构化中间层。
2. AI 处理流程必须拆成多段流水线，不能把所有任务塞进一个大 prompt。
3. 任务系统需要异步化，使用状态机管理解析、分析、生成、导出等步骤。
4. 输出结果必须能定位到原文，支持“原文 ↔ AI 分析 ↔ 测试用例”的关联展示。
5. 用户对 AI 结果的采纳、修改、拒绝需要沉淀为反馈数据，用于后续优化提示词、规则和评估集。

## 2. 核心功能范围

### 2.1 文档上传模块

支持能力：

1. 支持拖拽上传和手动选择文件。
2. 支持 Word、PDF、图片、HTML、Markdown、TXT 等格式。
3. 上传时校验文件大小、文件类型、页数限制和安全风险。
4. 上传后生成文档记录，并创建异步解析任务。

建议限制：

1. MVP 阶段优先支持 Markdown、TXT、Word、PDF。
2. 图片和扫描版 PDF 可以后置，依赖 OCR 能力。
3. HTML 解析需要做正文抽取，避免导航、广告、脚本内容干扰 AI。

### 2.2 文档解析与结构化中间层

文档解析后不要直接把全文丢给 AI，而是先生成统一结构。

建议结构如下：

```json
{
  "documentId": "doc_001",
  "title": "需求文档标题",
  "sourceType": "pdf",
  "sections": [
    {
      "sectionId": "sec_001",
      "title": "登录功能",
      "level": 1,
      "text": "用户可以通过手机号和验证码登录...",
      "pageRange": [1, 2],
      "position": {
        "startOffset": 0,
        "endOffset": 120
      },
      "tables": [],
      "images": [],
      "sourceSnapshot": {
        "rawCharLength": 120,
        "whitespaceRatio": 0.08,
        "tableMarkerDensity": 0.0,
        "specialCharDensity": 1.2,
        "alphanumericRatio": 0.42,
        "lineCount": 3
      },
      "parseConfidence": 0.92
    }
  ],
  "metadata": {
    "fileName": "prd.pdf",
    "createdAt": "2026-06-29",
    "parserVersion": "v1",
    "documentParseConfidence": 0.88
  }
}
```

中间层必须包含：

1. 原文内容。
2. 标题层级。
3. 表格、图片、列表等结构信息。
4. 页码、段落、字符偏移等定位信息。
5. 解析置信度和异常信息，例如 OCR 低置信度、表格解析失败。
6. **原文快照（sourceSnapshot）**：除解析后的 `text` 外，记录该段落的原始字符长度、空白占比、表格符号密度（如 `|`、制表符）、特殊字符密度等启发式指标，用于事后评估解析质量。

这样后续才能做到点击 AI 问题后定位原文、高亮段落、回溯生成依据。

#### 2.2.1 原文快照与解析质量兜底

仅记录 `parserVersion` 不足以在 AI 输出异常时主动发现问题。建议在每个 section 写入 **sourceSnapshot**，并在分析完成后做交叉校验：

| 快照字段 | 说明 |
|----------|------|
| `rawCharLength` | 段落原始字符长度 |
| `whitespaceRatio` | 空白字符占比 |
| `tableMarkerDensity` | 每百字符表格符号数（`\|`、制表符等） |
| `specialCharDensity` | 每百字符非常规特殊字符数 |
| `alphanumericRatio` | 字母数字占比（过低常见于 OCR 乱码） |
| `parseConfidence` | 由快照启发式计算的单段置信度 0~1 |

**主动告警规则**（后端确定性逻辑，不依赖 LLM）：

1. 文档级 `documentParseConfidence`（按段落长度加权）低于阈值（建议 0.65）。
2. AI 问题分析结果中 `evidence_type=inferred` 或空 `source_refs` 占比过高（建议 ≥ 45%）。
3. `source_refs.quote` 无法在对应 section 正文中匹配（引用错乱）。

当同时命中解析低置信 + 引用异常时，系统应向前端/任务面板推送告警，例如：

> 文档解析置信度低于阈值，可能影响分析质量，建议检查 PDF 是否为扫描件或版式是否复杂。

该机制比单纯记录 `parserVersion` 更主动，可在用户质疑 AI 结论之前提示「输入质量风险」。

### 2.3 需求问题分析模块

AI 分析不要一次完成，建议拆成多个子任务：

1. 需求摘要：提取业务目标、用户角色、核心流程、关键规则。
2. 需求拆分：将文档拆成原子需求点，每个需求点带来源位置。
3. 完整性检查：检查是否缺少前置条件、异常流程、边界条件、权限规则、数据规则、验收标准。
4. 一致性检查：识别前后矛盾、术语不一致、状态流转冲突、规则重复。
5. 清晰度检查：识别模糊词，例如“尽快”“合理”“支持多种”“默认”等，并要求补充可验证标准。
6. 风险评级：按影响范围、阻塞程度、修复成本给问题分级。
7. 结论整合：合并重复问题，生成可读报告。

建议输出字段：

```json
{
  "issueId": "issue_001",
  "type": "unclear | missing | conflict | risk",
  "severity": "high | medium | low",
  "title": "登录失败次数限制不明确",
  "description": "文档提到登录失败需要限制，但未说明失败次数、锁定时长和解锁方式。",
  "sourceRefs": [
    {
      "sectionId": "sec_003",
      "quote": "多次登录失败后限制登录",
      "page": 4
    }
  ],
  "suggestion": "建议明确失败次数、锁定时间、是否支持管理员解锁、是否区分账号和设备。",
  "status": "pending"
}
```

### 2.4 测试文档生成模块

测试用例生成应该基于“结构化需求点 + 需求问题分析结果”，而不是直接基于原文。

生成类型：

1. 功能测试用例。
2. UAT 验收用例。
3. 异常场景用例。
4. 边界值用例。
5. 权限与角色用例。
6. 数据校验用例。
7. 流程类用例。

建议测试用例结构：

```json
{
  "caseId": "TC_001",
  "module": "登录",
  "title": "手机号验证码登录成功",
  "priority": "P0",
  "precondition": "用户手机号已注册且账号状态正常",
  "steps": [
    "打开登录页",
    "输入已注册手机号",
    "输入正确验证码",
    "点击登录"
  ],
  "expectedResult": "登录成功并进入首页",
  "caseType": "functional",
  "sourceRequirementIds": ["req_001"],
  "riskNotes": []
}
```

导出格式：

1. Excel：适合测试执行和评审。
2. Markdown：适合版本管理和协作。
3. XMind：适合脑图式测试设计和评审展示。
4. JSON：作为系统内部结构化结果，便于二次处理。

### 2.5 任务管理面板

页面需要展示所有任务及其状态。

建议状态机：

1. `created`：任务已创建。
2. `uploaded`：文件已上传。
3. `parsing`：文档解析中。
4. `parsed`：解析完成。
5. `analyzing`：需求分析中。
6. `analysis_completed`：分析完成。
7. `generating_cases`：测试用例生成中。
8. `case_completed`：测试用例生成完成。
9. `exporting`：导出中。
10. `completed`：任务全部完成。
11. `failed`：任务失败（见下方触发条件）。
12. `cancelled`：任务取消。

**`failed` 触发条件**（满足任一即转入，并写入 `error_code` / `error_message`）：

1. **步骤超时（`step_timeout`）**：当前步骤执行时间超过该步骤配置的 `timeout_seconds`（AI 调用因网络或模型负载挂起是生产环境常见原因）。
2. **重试耗尽（`max_retries_exceeded`）**：步骤失败且 `retry_count >= max_retry`。
3. **不可恢复错误**：如文件损坏、JSON Schema 校验在最大重试后仍失败、存储不可用等。

每个状态建议记录：

1. 当前进度百分比。
2. 当前步骤说明。
3. 开始时间、结束时间。
4. 错误码、错误详情（超时失败时 `error_code` 建议为 `step_timeout` 或 `ai_call_timeout`）。
5. **步骤级 `timeout_seconds` 与 `max_retry`**（任务表存默认值，步骤表可覆盖）。
6. 已重试次数 `retry_count` / 最大重试 `max_retry`。
6. **`quality_warnings`（JSON）**：质量告警清单，供任务详情页醒目展示。当解析置信度低、AI 大量 inferred、引用错乱或用例覆盖率不足时由后端自动写入，示例：

```json
{
  "items": [
    {
      "warning_type": "parse_quality",
      "level": "warning",
      "message": "文档解析置信度低于阈值，可能影响分析质量，建议检查 PDF 是否为扫描件或版式是否复杂。",
      "metrics": {
        "document_parse_confidence": 0.52,
        "inferred_issue_ratio": 0.48,
        "broken_citation_ratio": 0.25
      }
    }
  ],
  "alert_level": "warning",
  "should_warn_user": true,
  "updated_at": "2026-07-06T18:00:00"
}
```

任务详情页应在 `should_warn_user=true` 时顶部展示告警条，并引导用户检查原始文档或重新上传。

**MVP API 接口（已实现）**：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/tasks` | 任务列表 |
| GET | `/api/tasks/{task_no}` | 任务详情（含 quality_warnings） |
| GET | `/api/tasks/{task_no}/report` | 分析报告 |
| POST | `/api/tasks/run` | 上传文档并异步执行流水线 |

#### 2.5.1 步骤超时与重试（生产必备）

设计任务状态流转时，**每个步骤**（含 AI 子步骤）应配置：

| 配置项 | 说明 | MVP 建议默认值 |
|--------|------|----------------|
| `timeout_seconds` | 单步最大执行时长，超时则标记步骤 `failed` 并触发重试或任务 `failed` | 见下表 |
| `max_retry` | 该步骤最大重试次数（不含首次执行） | 2~3（AI 步骤 3） |
| `retry_count` | 当前已重试次数 | 0 起计 |

**建议超时（可按步骤覆盖）**：

| 步骤 | timeout_seconds | max_retry | 备注 |
|------|-----------------|-----------|------|
| `parse` / `clean` / `chunk` | 300 | 2 | 大文件解析 |
| `extract_requirements` | 120 | 3 | 单次 LLM 调用，对齐 `AI_TIMEOUT_SECONDS` |
| `check_requirement_completeness` | 90 | 2 | 反思步骤，通常较快 |
| `analyze_requirements` | 180 | 3 | 问题分析 |
| `generate_cases` | 180 | 3 | 用例生成 |
| `validate` / `export` | 60 | 2 | 本地校验与导出 |

**调度与轮询约定**：

1. 异步任务先用数据库状态轮询时，Worker 或定时任务需检测 `running` 步骤是否已超时（`now - started_at > timeout_seconds`）。
2. 超时未响应的 AI 请求应主动 cancel（若 SDK 支持），避免僵尸步骤占用并发。
3. 重试应**仅重跑失败步骤**（断点续跑），并递增 `retry_count`；达到 `max_retry` 后将任务置为 `failed`。
4. `nb_test_task_steps` 与 `nb_test_ai_runs` 均记录 `duration_ms`，便于后续调优超时阈值。

### 2.6 分析报告展示区

这是产品交互核心。

建议采用三栏布局：

1. 左侧：文档目录和原文内容。
2. 中间：AI 发现的问题列表，支持按严重程度、问题类型、模块筛选。
3. 右侧：问题详情、修改建议、用户操作区。

关键交互：

1. 点击问题后，高亮原文对应段落。
2. 点击原文段落后，展示关联的问题和测试用例。
3. 支持“采纳”“修改”“拒绝”“标记为误报”。
4. 支持添加人工备注。
5. 支持将某个问题转为待补充需求。
6. 支持导出需求问题清单。

### 2.7 AI 结果反馈闭环

用户操作需要沉淀为反馈数据。

反馈类型：

1. 用户采纳了某条问题。
2. 用户修改了 AI 的描述或建议。
3. 用户拒绝了问题，并标记拒绝原因。
4. 用户新增了 AI 没发现的问题。
5. 用户修改了测试用例的步骤、预期结果或优先级。

这些数据后续可用于：

1. 优化 prompt。
2. 建立评估集。
3. 统计 AI 命中率和误报率。
4. 训练团队内部的需求检查规则库。

## 3. 系统架构建议

建议拆成以下服务或模块：

1. 前端 Web：上传文档、任务面板、报告展示、测试用例预览、导出操作。
2. 后端 API：用户、文档、任务、报告、用例、导出等业务接口。
3. 文件存储：保存原始文档、解析产物、导出文件。
4. 文档解析服务：负责 Word、PDF、HTML、图片 OCR 解析。
5. AI 编排服务：负责 prompt 模板、分段调用、结果校验、重试。
6. 异步任务队列：负责长耗时任务调度。
7. 数据库：保存结构化文档、任务状态、分析结果、测试用例、反馈记录。

MVP 阶段可以先做成单体后端，但内部代码仍建议按模块拆分，避免后期重构成本过高。

## 4. AI 流水线设计

建议流程：

1. 文档解析：原始文件转结构化中间层。
2. 内容清洗：去除页眉页脚、重复目录、无效空白、广告脚本等。
3. 分块切片：按标题、模块、语义边界切分，保留上下文引用。
4. 需求点抽取：从每个章节提取原子需求。
5. **需求点完备性自检**（反思步骤）：基于文档标题、目录层级和已抽取结果，让 AI 自我审视覆盖是否完整。
6. 需求关系建模：识别角色、流程、状态、规则、数据对象。
7. 问题检查：分别执行清晰度、完整性、一致性、可测试性检查。
8. 问题合并：去重、合并相似项、统一严重程度。
9. 测试设计：根据需求点生成测试点和测试场景。
10. 用例生成：生成结构化测试用例。
11. 结果校验：检查用例是否缺少步骤、预期结果、来源需求。
12. 导出：输出 Excel、Markdown、XMind。

### 4.1 隐式依赖与完备性自检

流水线中存在一条关键但易被忽视的依赖链：**第 7 步（问题检查）强烈依赖第 4 步（需求点抽取）和第 6 步（关系建模）的质量**。若需求点抽取不完整，后续针对单个需求点的「完整性检查」「清晰度检查」等子任务将因输入缺失而失效——AI 无法报告它从未「看到」的需求中的问题，召回率会显著下降。

因此在第 4 步与第 6 步之间增加 **第 5 步：需求点完备性自检**，作为典型的「反思（Reflection）」机制：

1. **输入**：文档标题、完整目录树（`sections` 层级）、第 4 步产出的原子需求列表及其 `sectionId` 关联。
2. **自检问题**（写入 Prompt，引导 AI 逐项回答）：
   - 基于文档标题和目录，我抽取的需求点是否覆盖了所有章节？
   - 是否存在整节/整段未被抽取的模块（如附录、异常流程、权限说明）？
   - 某些章节是否只抽到了摘要级描述，而遗漏了具体规则或验收标准？
3. **输出与处置**：
   - 产出「覆盖映射表」：每个 `sectionId` 对应 0~N 条需求点，并标记 `covered | partial | missing`。
   - 对 `partial` / `missing` 章节触发**定向补抽**（仅重跑对应章节的抽取，而非全量重跑第 4 步）。
   - 将自检结论写入中间结果，供第 7 步问题检查参考；若某章节经补抽仍为空，可主动生成一条「该章节可能缺少可测试需求描述」的提示型问题。

该步骤成本较低（通常一次轻量 LLM 调用 + 按需局部补抽），但能显著提升后续分析与用例生成的召回率，应在 `nb_test_task_steps` 中独立记录，支持单独重试。

每一步都应保存中间结果，便于失败重试、调试 prompt 和追踪质量。

## 5. 质量控制与防幻觉设计

AI 输出需要被约束。

建议策略：

1. 所有问题必须引用原文片段或说明“未在文档中找到依据”。
2. 禁止输出没有来源的确定性结论。
3. 对“推测型建议”和“原文明确问题”做区分。
4. 输出必须符合 JSON Schema，后端校验不通过则要求 AI 重试修复。
5. 对高风险结论要求二次校验。
6. **需求点完备性自检**：在关系建模与问题检查之前，对照文档目录校验抽取覆盖率，对遗漏章节定向补抽（见 §4.1）。
7. 对测试用例进行覆盖率检查，例如每个原子需求至少关联一条用例。
8. 保留 AI 调用日志、输入摘要、输出结果和模型版本。
9. **解析质量兜底告警**：基于 section 原文快照 + AI 引用校验，在 inferred 占比过高或引用错乱时自动提示用户检查文档解析质量（见 §2.2.1）。

## 6. 数据模型草案

> 表命名规范：所有业务表统一使用 `nb_test_` 前缀，详细字段定义见 [DB表.md](./DB表.md)。

核心表或集合：

1. `nb_test_documents`：文档基础信息。
2. `nb_test_document_sections`：解析后的章节和段落。
3. `nb_test_tasks`：异步任务状态。
4. `nb_test_task_steps`：任务流水线步骤明细。
5. `nb_test_requirements`：抽取后的原子需求。
6. `nb_test_requirement_issues`：需求问题。
7. `nb_test_issue_source_refs`：问题与原文位置的引用关系。
8. `nb_test_test_cases`：测试用例。
9. `nb_test_test_case_requirements`：测试用例与需求点多对多关联。
10. `nb_test_exports`：导出文件记录。
11. `nb_test_feedbacks`：用户反馈。
12. `nb_test_ai_runs`：AI 调用记录。
13. `nb_test_prompt_templates`：Prompt 模板版本管理。
14. `nb_test_users`：用户（团队化阶段）。

关键关系：

1. 一个文档（`nb_test_documents`）对应多个章节（`nb_test_document_sections`）。
2. 一个章节对应多个原子需求（`nb_test_requirements`）。
3. 一个原子需求对应多个问题（`nb_test_requirement_issues`）和多个测试用例（`nb_test_test_cases`）。
4. 一个测试用例通过 `nb_test_test_case_requirements` 关联多个原子需求。
5. 用户反馈（`nb_test_feedbacks`）关联问题、用例或 AI 运行结果（`nb_test_ai_runs`）。

## 7. 前端页面设计

建议页面：

1. 首页 / 任务列表：展示最近任务、状态、创建时间、操作入口。
2. 文档上传页：上传文件、选择分析类型、配置输出格式。
3. 任务详情页：展示进度、日志、失败原因、重试按钮。
4. 需求分析报告页：展示原文、问题列表、问题详情。
5. 测试用例页：表格预览、筛选、编辑、批量导出。
6. 设置页：配置模型、prompt 模板、导出模板、规则库。

MVP 阶段可以先合并页面：

1. 上传 + 任务列表。
2. 任务详情 + 分析报告。
3. 测试用例预览 + 导出。

## 8. 分阶段落地计划

### 阶段一：跑通最小闭环

目标：验证 AI 分析和测试用例生成是否可用。

范围：

1. 支持 Markdown / TXT / Word 文档上传。
2. 将文档解析为纯文本和简单章节结构。
3. 调用 AI 生成需求问题报告。
4. 调用 AI 生成 Markdown 格式测试用例。
5. 前端支持上传、查看任务状态、查看结果、下载 Markdown。

### 阶段二：结构化与可定位

目标：提升可用性和可信度。

范围：

1. 完善结构化中间层。
2. 增加原文定位和高亮。
3. 问题结果支持类型、严重程度、来源引用。
4. 测试用例关联需求点。
5. 支持 Excel 导出。

### 阶段三：多格式与交互闭环

目标：支持更多真实业务文档。

范围：

1. 支持 PDF、HTML、图片 OCR。
2. 支持用户采纳、修改、拒绝 AI 结果。
3. 建立反馈数据。
4. 增加规则库和 prompt 模板管理。
5. 支持 XMind 导出。

### 阶段四：质量评估与团队化使用

目标：让系统具备稳定可运营能力。

范围：

1. 建立 AI 输出质量指标。
2. 建立需求分析评估集。
3. 支持团队空间、权限、历史版本。
4. 支持多模型对比。
5. 支持与项目管理、测试管理平台集成。

## 9. 可以继续改善的点

1. 增加“需求可测试性评分”：从完整性、清晰度、一致性、可验证性四个维度给文档评分。
2. 增加“追问清单”：AI 不仅指出问题，还生成可直接发给产品经理的问题列表。
3. 增加“需求变更对比”：上传新旧版本文档后，识别需求变化并提示需要新增或回归的测试用例。
4. 增加“规则库”：沉淀团队常见检查规则，例如权限、金额、状态流、通知、数据导出等。
5. 增加“模板化导出”：不同团队可以配置 Excel 字段、用例命名规则、优先级规则。
6. 增加“覆盖率视图”：展示需求点是否都有测试用例覆盖，哪些需求存在风险。
7. 增加“人工编辑后再导出”：避免 AI 结果未经确认直接进入正式测试资产。
8. 增加“失败重试与断点续跑”：某一步失败时只重跑该步骤，不重跑整个任务。
9. 增加“成本控制”：记录每个任务的 token 消耗、模型费用和耗时。
10. 增加“隐私与脱敏”：对手机号、邮箱、身份证、客户名称等敏感信息做脱敏处理。

## 10. MVP 技术实现建议

前端建议：

1. 使用 React / Vue 均可，优先选择团队熟悉技术栈。
2. 上传组件支持拖拽、进度条、格式校验。
3. 任务状态通过轮询实现，后续再升级 WebSocket。
4. 测试用例预览先用表格组件实现。

后端建议：

1. 提供 REST API。
2. 文件上传后保存到本地或对象存储。
3. 异步任务可先用数据库状态轮询或简单队列实现；**务必为每个步骤配置 `timeout_seconds` 与 `max_retry`**（见 §2.5.1）。AI 接口因网络或模型负载导致超时是生产环境常态，`failed` 状态须明确包含「步骤超时」与「重试耗尽」两类触发条件，并写入 `error_code`（如 `step_timeout`、`ai_call_timeout`）。
4. AI 调用统一封装为 `AiService`，不要散落在业务代码中；客户端请求超时与步骤级超时分层：HTTP 客户端 timeout ≤ 步骤 timeout。
5. Prompt 模板独立存储，便于调试和版本管理。
6. **多模型混合路由**：不同流水线步骤选用不同模型，在 `ModelRouter` 中按 `RunType` 与输入规模自动选择（见 §10.1）。

#### 10.1 多模型混合路由

单一模型难以同时满足「复杂推理、结构化生成、低成本格式化、超长上下文」四类需求。建议按任务类型配置不同模型，由 `AiService` 在每次调用前通过 `ModelRouter` 解析 `ModelProfile`：

| 任务类型 | RunType / 步骤 | 推荐模型 | 原因 |
|----------|----------------|----------|------|
| 复杂需求分析 | `extract_requirements` / `analyze_requirements` | GPT-4-turbo / Kimi | 理解能力强，能处理复杂逻辑与矛盾检测 |
| 测试用例生成 | `generate_cases` | GPT-4o / 智谱 GLM-4 | 结构化 JSON 输出稳定，质量与成本平衡 |
| 格式转换 / 轻量反思 | `check_requirement_completeness` | GPT-3.5-turbo | 目录对照与 JSON 格式化，成本低、速度快 |
| 大规模文档处理 | `extract_requirements`（字符数 ≥ 阈值） | Kimi (32k) / 通义千问 | 支持超长上下文，减少截断遗漏 |

**路由规则**：

1. 环境变量 `AI_MULTI_MODEL_ENABLED=true` 时启用；为 `false` 则全程使用 `AI_MODEL`。
2. 各分类可独立配置 `AI_MODEL_*` / `AI_BASE_URL_*` / `AI_API_KEY_*`；未配置时回退到默认 `AI_*`。
3. 当文档总字符数 ≥ `AI_LONG_CONTEXT_THRESHOLD_CHARS`（默认 20000）时，`extract_requirements` 自动切换为 `long_context` 模型。
4. 每次 AI 调用在 `nb_test_ai_runs.input_summary` 记录 `model_category` 与 `model`，便于成本核算与效果对比。

查询当前路由：`GET /api/models/routing`。

文档解析建议：

1. Word：使用 `mammoth` 或同类库提取结构化文本。
2. PDF：优先处理文本型 PDF，扫描件放到 OCR 阶段。
3. HTML：使用正文抽取和标签清洗。
4. 图片：使用 OCR 服务，结果必须带置信度。

导出建议：

1. Excel：后端生成 `.xlsx` 文件。
2. Markdown：直接由结构化测试用例渲染。
3. XMind：可先输出兼容 XMind 的 Markdown 层级，再进一步生成原生格式。

## 11. 关键风险

1. 文档解析质量不足会直接影响 AI 输出质量。**兜底**：section 级原文快照 + 分析后引用校验，低置信或大量 inferred/引用错乱时主动告警（见 §2.2.1）。
2. 长文档 token 成本和耗时较高，需要分块和缓存。
3. **AI 调用超时与模型负载**：网络抖动或网关排队可能导致 LLM 调用长时间无响应；必须为步骤设定 timeout 与 max_retry，避免任务永久卡在 `analyzing` 等中间态（见 §2.5.1）。
4. AI 可能产生没有原文依据的结论，需要来源引用和校验机制。
5. 多格式文档支持容易拖慢 MVP，建议分阶段支持。
6. 如果没有用户反馈闭环，系统很难持续变准。
7. 测试用例生成质量依赖需求结构化程度，不能只依赖一个 prompt。
8. **流水线隐式依赖**：问题检查（第 7 步）的上游质量由需求点抽取（第 4 步）和关系建模（第 6 步）决定；若跳过完备性自检，遗漏章节上的缺陷将无法被后续步骤发现。

## 12. 推荐优先级

优先做：

1. 文档上传。
2. Word / Markdown / TXT 解析。
3. 结构化中间层。
4. AI 需求问题分析。
5. AI 测试用例生成。
6. Markdown / Excel 导出。
7. 任务状态机。

暂缓做：

1. 图片 OCR。
2. 复杂 PDF 版式还原。
3. 多模型对比。
4. 团队权限。
5. 外部平台集成。
6. 原生 XMind 格式深度适配。
